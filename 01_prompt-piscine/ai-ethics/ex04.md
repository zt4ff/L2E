# AI as Learning Amplifier

## Phase 1: Build Foundation (me first)

**Active reading + self-explanation**<br>
From what I have read before: Wireless routing protocols in constrained environments (like IoT / sensor nets) differ from classic ones (OSPF, BGP) because nodes have very low power, lossy links, limited memory/CPU, and often many-to-one traffic (sensors -> sink/gateway).

### Key protocols that come to mind for wireless / low-power scenarios:

- **RPL** (Routing Protocol for Low-Power and Lossy Networks): The IETF standard for LLNs (Low-power Lossy Networks). Distance-vector-ish but builds Destination-Oriented Directed Acyclic Graphs (DODAGs). Proactive-ish with trickle timers to reduce control messages. Supports objective functions (e.g., minimize ETX for reliability or energy). Handles upward (many-to-one), downward, and point-to-point traffic via modes of operation.

**Simple scenario design (7 devices)**
Imagine a small indoor office sensor deployment (temperature + occupancy monitoring):

- 1 border router (sink, always powered, connected to internet)
- 6 battery-powered sensor nodes arranged roughly in two rows of 3

Topology: chain-like but with a couple of cross links (not fully mesh, some nodes 2 hops from sink).

### My justification for this small scenario

I'd pick RPL here.
Why?

- Mostly static nodes → proactive route maintenance is ok.
- Traffic pattern = mostly upward (sensors report periodically to sink) → RPL's default DODAG root at sink fits perfectly.
- Trickle timers drastically reduce control packet overhead compared to AODV flooding or OLSR periodic hellos.
- Objective function can prioritize link reliability (ETX) over hop count → important in indoor environments with interference.

AODV would waste energy on route discoveries every time a periodic report needs to go out if routes time out. OLSR's periodic topology control would drain batteries faster in this low-traffic but always-on sensing setup.

## Phase 2: Strategic AI Use

I think I have a decent grasp, but let's probe weak spots and edge cases I’m unsure about.

Questions I would ask an AI (and what I learned / confirmed):

- In a network with occasional mobility (e.g., one node on a moving robot), how badly does RPL degrade vs AODV?
  → RPL struggles with mobility because DODAG repairs are slow and expensive (global repair can flood the network). AODV handles mobility better because routes are discovered on-demand and invalidated quickly on failure. For mostly static + rare mobile nodes → RPL with local repair is often still preferred, but pure mobile → AODV or hybrids.
- What's the main failure mode of RPL in very dense networks (hundreds of nodes in same radio range)? → Excessive parent switching / rank churn if ETX estimates fluctuate, leading to control packet storms and energy drain. Trickle timers help but don't eliminate. Also, downward routes can be unreliable without DAO-ACKs enabled.
- Is there a real reason to still use AODV or OLSR in modern IoT deployments?
  → Rarely. RPL is dominant in 6LoWPAN / Thread / Contiki-NG / RIOT implementations. AODV derivatives (LOADng) exist for very specific cases needing better P2P support. OLSR is almost never used in serious low-power IoT.

Validation: My initial choice of RPL for static sensor net seems solid literature points to RPL being more power-efficient than AODV/OLSR in LLNs.

## Phase 3: Real Application — Small smart-city network

Design:

- 1,000 IoT sensors (air quality, noise, parking, flood, etc.) — mostly static, battery/solar, low data rate (1 msg every 5–60 min)
- 50 traffic lights / controllers — line-powered, need reliable bidirectional comms (status up + commands down)
- 10 emergency vehicles — mobile, need low-latency vehicle-to-infrastructure (V2I) for priority signaling

Assume heterogeneous radio: most sensors use 802.15.4 / 6LoWPAN (short range, low power), traffic lights maybe on the same or Wi-Fi/5G backhaul, vehicles likely DSRC / C-V2X but we simplify to include them in routing domain.

### Protocol decisions & justification:

- Core protocol for sensors + traffic lights: RPL over 6LoWPAN
  Justify: Standard for LLNs, scales to thousands of nodes, energy-efficient for periodic upward traffic from sensors, supports downward for actuators (traffic lights). Objective function = OF0 (hop count) for simplicity or ETX/MRHOF for better reliability in urban interference. Multiple DODAGs possible if we need to separate domains (e.g., one for environmental sensors, one for traffic control).

- For emergency vehicles (mobile): Hybrid / fallback to AODV-like on-demand routing or a separate overlay
  Justify: RPL is poor for fast-moving nodes (rank instability, slow convergence). Use RPL for static infrastructure, but vehicles use reactive discovery (AODV/LOADng) when they need to route through the mesh to reach a specific traffic light. Or — more realistically — vehicles connect via cellular/5G to a central cloud that commands traffic lights, bypassing the low-power mesh entirely.

- Overall architecture: Multi-sink / multiple border routers (gateways placed strategically across the city district) running RPL with one root per gateway. Gateways aggregate and forward to cloud via fiber / 5G.

### Failure points / risks:

- RPL rank attacks / version number attacks → malicious node advertises fake low rank → blackhole / sinkhole.
- Energy hotspots around popular parents / near gateways.
- Downward traffic unreliability if nodes don't maintain DAO state properly.
- Mobility-induced partitions or long convergence times when emergency vehicle moves fast.
- Interference in dense urban 2.4 GHz band → high churn in preferred parent selection.
- Scalability limit around ~1,000–2,000 nodes per DODAG before control traffic explodes.

### Refinement after AI feedback (simulated):

- Add secure mode (RPL with certificates / signatures) or use RPL-SEC extensions to mitigate attacks.
- Implement trickle++ or adaptive timers to dampen churn.
- Use mobility-aware variants (e.g., mRPL) or hybrid RPL + reactive for vehicles.
  Split into smaller DODAGs per neighborhood to improve scalability.
